Why Software Engingeering?

March 27, 2022 by Isabella

Programming has been a skill on my bucket list since I could remember. I admire people who has the know-how, so much so I’ve married to a programer.

I had a few attempts at learning to code in school ages, however found extremely challenging getting my head around it and applying what I’ve learnt to solve real world problems. During my university years, I’ve even changed my major from Electrical/electronics to Mechatronics halfway through the degree because I’m struggling with courses like Embeded System. But why again and why now?

Ability to create

Able to design, create and improve something is a powerful capability and something I desire. This is why I’ve studied engineering. Unfortunately with the decline of Australian manufacturing, it is very rare to get a job where you can actually engineer a product. Most of people like myself, after graduate got into a technical role and acting as the knowledge link between oversea’s engineers and local technicians.

However, making something my own has always been in my hart. I’m hoping giving learn to code another shot will allow me one day able to just sit down and write an application. No longer stuck just wondering what could be and what if.

Highly transferable skill

Been a manager myself, when hiring for an experienced position, I would often only interview those with relevant experiences. Have you worked in the same industry? Perhaps even on products with similar technical attributes?

Unlike many other disciplines, programming is a much transferable skill, which means boarder career options with wider range of industries. Because it is also in high demand the fear of never getting a job is nonexistent. Worse comes to worst, you can always find project based contract works.

Career progression & benefits

My day are filled with meetings, handling escalations (and only escalations) and coaching others. Higher pay often comes with additional responsibilities and sacrifices in personal life. While I’m already earning in six-figures and happy with my package (for now), it is still significantly lower than my programer husband. We are young parents with a three year old. It bothers me time to time my husband is able to spend more time with my son.

Looking around, become a programer seems like a great alternative. After gaining a couple of years field experiences, it is highly likely to land a midlevel job, getting the same pay as I’m currently on without been in a management role. To tip it up, full remote and flexibility is no longer a dream.

Phase 0 Portfolio Project Reflection

April 3, 2022 by Isabella

Now I’ve started a software engineering bootcamp course with Academy Xi, the first project we have at the end of Phase 0 is to host a personal website on GitHub. Apart from the project, we were also required to publish two blogs sharing the reasons of studying software engineering, as well as any reflection had during this initial phase and on the project of building this website. I then thought to myself, what is better then combine them all and build a website to host those blogs. What you are about to read are my reflections of the experience so far.

1) What went well? Key Learning.

In the last three weeks, we’ve completed a number of labs, got introduced to GitHub and the basics of HTML and CSS. Everything went quite smooth when following guidance provided, initialising repositories and creating basic HTML, CSS and JavaScript files.

Having a husband working in this field definitly helps. He suggested me to start my website utilising free front-end open source toolkit such as Bootstrap. Following steps below, I was able to have a jump start at laying out my website.

  • Added meta tags for better mobile experience to <head>.
  • Linked Bootstrap CCS stylesheet and followed by a local style.css file, which gives me the flexibility of applying existing stylings from Bootstrap and override with mines when needed.
  • Started HTML content from a Magazine line blog template on Bootstrap by inspecting its elements.
2) What could have been improved?

Evidently writing blog content directly into a HTML file, especically with the index.html isn’t user friendly. Even when the only user is me, it is definitely not my prefered long term solution. Hoping as I progress with the course, rather than editing codes directly, my website will include blog templates and back-end server to allow posting new blogs directly onto the site.

On top of that, as there is only one HTML file, most of the links on the site are going nowhere. Will need to work on enrich contents on the website. A few ideas to start with are:

  • Store actual blogs on individule pages and access them via links on featured posts or in archives section.
  • Expend the use of the website to store other files such as my CV, and add navigation bar to the header to access different contents.

Looking at the process alone, what could have been done better is to have smaller commits more frequently. After commit with bootstrap stylesheet, I got carried away with creating the content and adjusting styles using the opened HTML file. No additional commit were added until the initial site was pretty much finished.

3) The key take aways.

Here are some key learnings from this eventful initial phase.

  • Read, I mean READ error messages. The fix to the error is often in the message.
  • Do not get caught on what people were telling you to do. There are many good ways at solving a problem. I got stuck in one of the lab where Column structure were suggested. It was until I decided to use Grid layout after many days of trials and failures with Columns, I finally had something I’m happy with.
  • Smaller and more frequent commits.
  • Utilise tools and learn from other’s codes. Bootstrap is powerful, so is inspecting website codes.
4) The process you followed?

Step by step instructions provided by Academy Xi in this course has been very guiding the overall process of completing this project.

  1. Local repository folder and remote GitHub repository were created.
  2. Connected remote to the local repository and created README.md.
  3. Added basic index.html, style.css and script.js files.
  4. Published the website via GitHub Pages.
  5. Connected Bootstrap stylesheet & scripts.
  6. Built HTML layout utilising blog example on Bootstrap and customised components.
  7. Overrode certain styles.
  8. Finally. adding blogs like NOW!

Phase 1 Portfolio Project Reflection

May 29, 2022 by Isabella

Here comes the my second project of the software engineering bootcamp, we’ve been challenged to build a single page application to allow interactions with data from a public API. During initial planning, I had two ideas with similar concept, where the web application will take a location, either return its weather details or put a pin on the map.

My great dreams had soon woken by hash realities. Upon investigating a number of weather and map APIs, I came to the realisation that all of them require geographic coordinates as input parameters. Consider my skill level and time constrains on the project, choices became:

  1. User to enter geo coordinates and the APP returns weather details or map location.This would be more aligned with my original vision, but requires high user effort to have the geo-coordinates before able to use the APP. Meaning more trouble for me when it comes to demo time. On top of that, properly displaying weather icons and maps would be a different level of challenge.
  2. User to look up a location by name and the APP returns its geographic coordinates.Nothing fancy, but satisfies the project requirements and can be built on in the future if I still want a map out of it.

The preference was clear and decision was quick. I went with the 2nd option and please welcome to my very first APP - Coordinates Finder. As you typing up a location name, it will come up with some locations matches/suggestions for you to select. Once selected, you can click search icon to find its coordinates.

1) What went well? Key Learning.

Took learnings from last project, I’ve again linked my html with Bootstrap’s stylesheet. But this time, I’ve also spent sometime looked into its forms docs and unitised it’s search icon. When it came to coding time, my frontend form was completed rather smoothly with very limited style overwrites required.

After created remote repository, instead of then create the local directory and link them together, this time, I simply used ‘git clone’. Works a treat!

2) What could have been improved?

If time allows, there are a couple of issues I’d like to fix.

  • After user selected the best match address, the APP would still update and provide a list of suggestions based on the address selected. Would be good to stop the ‘fetchSuggestion()’ function once an option is selected.
  • If when user types in a location, but presses enter or search icon without selecting any of suggested options, the APP would run into an Uncaught TypeError as the search key not matching with any of the stored results. However, nothing would happen in the frontend. The ideal outcome in such situation would be either an alert pops up asking user to select an option listed or return coordinates of 1st option when nothing is selected.

Style wise, a more obvious feedback indicating a search is completed would be nice. A randomly updated background image is also a cool idea.

3) The key take aways.

Some interesting findings through this project.

  • The ‘debugger’ statement and ‘console.log()’ have been useful in this project. In particular when inspecting fetched data from API.
  • When working with ‘keyup’ event and creating options, ‘replaceChild()’ should be used instead of ‘appendChild()’ otherwise you would end up a very long list of options.
  • To avoid getting confused with the forever growing variables, visualising functions as building blocks with input and output. Limiting too many global variables and concentrate on variables within a function really helps.
4) The process you followed?

For this project, I’ve again followed along the tips and guidelines on Canvas.

  1. Planed out features for my APP and sketched how I wanted the website to look like.
  2. Investigated some public API, went back to the sketch board and updated the plan.
  3. Created a remote repository on GitHub and created local repository using ‘git clone’.
  4. Added basic html, css and js files and connected Bootstrap stylesheet.
  5. Built with the user form in index.html and overrode some styles.
  6. Created an account with mapbox.com and obtained an access token.
  7. Wrote a fetch function to query geographic data from Mapbox Geocoding API.
  8. Displayed address suggestions and stored suggestions in a result object.
  9. Finally, allowed searching in result and output geo-coordinates.

Phase 2 Portfolio Project Reflection

July 17, 2022 by Isabella

We’ve now been introduced to React. Same as the Javascript phase, we’ve again been assigned a project to build a single page application using data from an external API, however this time using React.

Instead of working on a new idea, I’ve decided to again build a Coordinate Finder APP same as last project with some additional features (listed).

  • Home/welcome route to access the APP.
  • Save function to post coordinates result to a JSON mock up backend server.
  • Record file route to view those saved places.
1) What went well? Key Learning.

As an improvement from last project, a randomly updated background image been added to this APP. To ensure contents are clearly displayed over the background image, a white transparent ‘Card’ was added as a container for the body of the APP.

Yarn was used for this project and turned out to be a great replacement of NPM. It performs much faster when comes to package installations, with no security vulnerability warnings like when using NPM.

I’ve also been introduced to a useful Chrome DevTools extension for React, the React Developer Tools. Its Components tab shows all components rendered in a tree with their current props and state.

2) What could have been improved?

At this stage, the APP and its mock up backend server are only running in development mode. Ideally would be good to deploy the APP for public use. Until then, others will need to fork the repository, then use ‘yarn start’ & ‘yarn json-server’ to run the APP. Alternatively for anyone who is interested to see the APP running, here is a demo video:

Function wise, the APP currently allows you to save empty records, which should’ve been rejected or ignored.

3) The key take aways.

Here are some key learnings from this project.

  • Previously discovered short cut where first create remote repository then git clone local directory, to avoid those steps of creating local directory and connecting with the remote GitHub repository, does not work when using create-react-app generator to create a new APP.
  • Always look for the latest documentations. For instance, with the use of react-router the latest documentation link provided is for dom v4, where ‘Switch’ component no longer work and was replaced by ‘Routes’ in its latest upgrade v6.
4) The process you followed?

Steps I went through creating this APP are:

  1. Setup basic file structure using create-react-app.
  2. Created a remote repository on GitHub and connected with the local repository.
  3. Installed react-router-dom and added Home & Main routes.
  4. Installed react-bootstrap, react-bootstrap-icons and imported Bootstrap stylesheet.
  5. Added basic html elements (header, navigation, forms & buttons) to Home and Main components.
  6. Created SearchForm and ResultForm child components in Main.
  7. Added a fetch function in SearchForm to query geographic data from Mapbox Geocoding API when changes are made to the search bar.
  8. Created DataList child component in SearchForm, rendering search suggestions from the fetched data.
  9. Update & pass state values back to Main to be accessed by ResultForm when SearchForm is submitted.
  10. Display geo-coordinates in ResultForm.
  11. Created a mock up backend using ’yarn add json-server --dev’ and posting places results when the Save button is clicked.
  12. Introduced a new SavedPlaces route to the App to display backend data in a table.
  13. Finally finished the App with some background styling.

Phase 3 Portfolio Project Reflection

September, 2022 by Isabella

We’ve been learning Ruby in this phase. While we need to again utilize data from an external API, project for this phase is asking to build Command Line Interface (CLI) application instead. You probably have guessed it, I’ve build the Coordinate Finder APP again using the same Mapbox Geocoding API.

Here is the link to the repository (CLI APP)

  1. The user will first be asked to enter a name or address to see a list of places matching that search text.
  2. User can then key a Place ID from that list to view the coordinates of that place.
1) What went well? Key Learning.

Without concerning much over the visual presentation, this project was completed rather quickly.

Prior to the project, Mentor walked us through a simple example, which helped as a jump start for my codings. Two key discoveries from that sessions are:

  • Test-driven development (TDD), i.e. adding APP feature begins by writing its test. It focus on user cases and requirements before code.
  • Postman, an API platform. While it has many features for building and using APIs, in this particular project, I’ve used Postman to view API responses, help setting my class attributes and arranging test data.
2) What could have been improved?

An error message has been included in the 2nd step, when the ID user entered does not match with any of the Place IDs listed, APP will return a message “Place ID not found!”.

However, it would be good if 1st step could also include some designed errors. At the moment when nothing matches with the search text, APP would return an empty list and still move to the next step asking to enter a Place ID. In certain scenarios, it will even return a long system error indicating the URI is invalid. Ideally, if there is no API response with the search text, a more descriptive error message should be displayed to instruct user to enter a different search text.

3) The key take aways.

Here are some key learnings from this project.

  • Class vs Instance methods: Class methods are always defined as “def self.method_name” whereas Instance methods do not come with the “self.”. By definition and in this project, Class methods are used to fetch & convert API responses into a collection of objects, i.e. places. Instance methods are then used to find and print those places instances.
  • break vs exit statements: exit statement stops the whole program while the break statement allows us to continue the rest of the program outside of the loop. In this project, break statement is used to go back to the place list search step.
4) The process you followed?

Steps I went through creating this APP are:

  1. Created a remote repository on GitHub and created local repository using ‘git clone’.
  2. Created Gemfile and installed gems.
  3. Configured the environment.
  4. Added basic APP features using TDD process, introduced Place class, method fetch API responses, method to construct a list of places and method to get place details.
  5. Created bin/places for user interface.
  6. Separated and refined classes, new classes Mapbox and PlaceCollections are introduced, as well as instance methods to find and print places.
  7. Added exit and go back features.
  8. Created README file with instructions on how to use the APP.
